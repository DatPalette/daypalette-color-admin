# 采样助手升级方案（admin-web 内部提案）

> 状态：proposal v0.1
> 日期：2026-05-08
> 适用范围：`apps/admin-web` 采样助手 / 候选审阅台下一阶段改版

## 1. 背景与结论

当前采样助手已经从“手工录入台账”转向“候选审阅台”，方向是对的，但现阶段仍存在两个根本问题：

1. **界面主视角不对。** 现在仍然以“逐条记录 + 单条详情”作为主工作面，颜色总览不够直观，运营无法先从全局看清当前批次的色路分布、重复情况和优先审核对象。
2. **数据来源不够可信。** 当前候选生成更多是“品牌模板 + 主题模板 + 搜索 URL 拼装 + 文本润色”，还不是基于真实服饰商品页的颜色采样，因此容易出现来源不清晰、色盘同质化、不同记录彼此重复的问题。

结论：

采样助手不适合继续在当前结构上做局部微调，建议升级为一套以**真实来源证据**、**全局色盘聚类**、**审核到上架闭环**为核心的新工作台。

---

## 2. 当前问题归纳

### 2.1 界面问题

当前界面的问题不只是“信息多”，而是**信息优先级反了**：

1. 运营最关心的是“这一批到底采到了哪些颜色、有哪些值得保留的色路、哪些明显重复”，但首页并没有先把色盘本身作为主角展示。
2. 现在需要点开一条记录，才看到一个颜色预览，这会导致工作流变成“逐条翻卡片”，而不是“先看全局，再挑重点审核”。
3. 批次信息、字段补录、修正表单占据了大量视觉注意力，但这些都不应该压过颜色本身。
4. 页面缺少按场景、季节、色路、可信度、重复风险的筛选和聚合能力，运营只能在线性列表里逐条处理。

### 2.2 数据问题

当前数据逻辑的问题集中在以下几点：

1. 候选来源不一定是真实商品页，部分只是搜索结果页或检索入口。
2. 页面上看不到“这组颜色是从哪个商品、哪个品牌、哪个页面、哪张图中得到的”完整证据链。
3. 颜色摘要更多是系统预设模板推演出来的，不是基于真实页面提色后的结构化结果。
4. 相似主题会不断产出相似颜色组合，缺少有效去重和聚类，导致同一批里出现多条“差不多”的色盘。
5. 当前数据合同把“候选记录”和“正式引用源字段”绑得太紧，导致证据层、聚类层、上架层没有被分开表达。

### 2.3 业务问题

对于面向中国 20 岁到 40 岁、偏年轻态女性用户的产品而言，当前结果还存在明显偏差：

1. 通勤路线容易过度收敛到安全的灰白蓝卡其，缺少更符合年轻女性气质的柔和提气色、轻都市感和场景变化。
2. 采样逻辑更像“找几个品牌凑候选”，还没有形成“围绕高频场景搜集受欢迎服饰配色”的系统方法。
3. 审核动作和上架动作之间的边界还不够清楚，运营难以判断“这条来源值得保留”与“这个色盘适合上架”是两个不同决策。

---

## 3. 升级目标

下一阶段采样助手应完成三个升级：

1. **采集升级**：从“模板候选生成”升级为“真实服饰来源证据采集 + 自动提色 + 去重聚类”。
2. **展示升级**：从“单条记录编辑”升级为“全局色盘总览 + 证据抽屉 + 聚类审核”。
3. **流程升级**：从“录一条、审一条”升级为“来源审核 -> 色盘聚类审核 -> palette 草稿 -> 上架审核”。

核心目标不是做一个更复杂的后台，而是让运营可以快速回答下面几个问题：

1. 这一批到底采到了哪些主流色路？
2. 这些颜色是不是来自可信的服饰页面？
3. 哪些色盘值得保留，哪些只是重复噪音？
4. 哪些已经足够支撑进入 palette 草稿和上架流程？

---

## 4. 目标用户与场景判断

### 4.1 目标用户

当前服务对象建议明确为：

中国 20 岁到 40 岁女性用户，整体偏年轻态，关注实穿、显气色、都市感和场景适配，不追求过度夸张或强实验性的高难度配色。

### 4.2 优先场景

建议采样助手优先服务以下高频场景，而不是继续只围绕单一 `workday` 场景扩容：

1. `workday`：上班通勤 / 轻正式办公
2. `city-weekend`：周末城市出行 / 咖啡馆 / 商场逛街
3. `light-social`：轻约会 / 聚餐 / 朋友见面
4. `holiday-outing`：短途出游 / 假日散步 / 城市旅行

### 4.3 优先关注的色路方向

针对该人群，建议优先采集以下高频方向，而不是让系统默认大量生成“安全中性色模板”：

1. 云感浅中性：奶白、壳白、燕麦、浅灰米
2. 雾蓝灰通勤：灰蓝、海雾蓝、钢蓝灰、雾石蓝
3. 奶茶驼系：奶驼、浅焦糖、灰卡其、暖褐灰
4. 柔和提气色：豆沙粉、雾粉杏、莓粉灰、藕粉灰
5. 低饱和绿灰：鼠尾草灰、浅橄榄灰、烟绿灰
6. 深色点缀：墨海军蓝、深板岩蓝、酒红棕、李紫灰

这些色路不应被系统写死为唯一答案，而应作为采样、聚类和审核时的参考标签体系。

## 5. 产品方向调整

### 5.1 从“记录审阅台”改为“色盘聚类审阅台”

新的采样助手建议把主工作对象从单条 `SamplingRecord` 改为“色盘聚类结果”。

原因：

1. 运营最终想审核的是“这组颜色值不值得保留”，而不是“这条记录字段填全了没有”。
2. 多个来源可能指向同一个高频色路，应该先聚成一组，再由人判断是否保留。
3. 去重和聚类之后，主理人处理的是几十个有代表性的色盘，而不是几百条碎片记录。

### 5.2 从“字段完整性”改为“证据可信度 + 色盘质量”

当前页面把“字段完整性”当作主判断标准，但下一阶段更重要的是：

1. 来源是不是官方或可信页面。
2. 颜色是不是从真实服饰页面提取出来的。
3. 色盘是不是有代表性、不是重复样本。
4. 是否能代表目标用户在某个场景中的高频受欢迎方向。

### 5.3 从“编辑表单”改为“筛选、比较、决策”

新工作台里，运营的核心动作应调整为：

1. 看总览
2. 筛选色路
3. 合并重复
4. 通过 / 驳回 / 暂缓
5. 转成 palette 草稿

手工补录字段仍然保留，但应退居二级操作，不再占据主界面中心。

## 6. 新的信息架构建议

### 6.1 首页：全局色盘总览

首页不再先展示记录列表，而是展示“当前批次的色盘墙”。每张卡片建议包含：

1. 4 到 6 个主色块预览
2. 色盘名称 / 临时标签
3. 对应场景
4. 季节或温感标签
5. 来源数量
6. 覆盖品牌数量
7. 来源可信度
8. 重复风险提示
9. 当前审核状态

首页顶部建议提供：

1. 场景筛选
2. 季节筛选
3. 色路标签筛选
4. 可信度筛选
5. 审核状态筛选
6. 重复风险筛选

### 6.2 右侧抽屉：证据详情

点开一张色盘卡片后，右侧抽屉展示对应证据，而不是跳进一整页表单。抽屉建议包含：

1. 真实来源页链接
2. 品牌与品类
3. 页面截图
4. 商品图缩略图
5. 提取颜色列表
6. 来源可信度说明
7. 与该聚类相近的相似色盘提示
8. 审核动作区

### 6.3 次级工作区：人工修正

只有在以下场景才进入人工修正：

1. 来源信息缺失但样本值得保留
2. 自动聚类错误，需要手工拆分或合并
3. 自动命名不准确，需要人工重命名
4. 需要补充运营备注和上架建议

人工修正不应该再是默认主流程。

## 7. 数据结构重组建议

当前建议把原本单层的采样记录结构拆成三层：

### 7.1 `SourceEvidence`

表示一条真实来源证据，核心字段建议包括：

1. `evidenceId`
2. `batchId`
3. `occasionId`
4. `brandName`
5. `platform`
6. `channelType`
7. `sourceUrl`
8. `pageTitle`
9. `observedAt`
10. `itemCategory`
11. `imageUrls[]`
12. `pageScreenshotUrl`
13. `extractedColors[]`
14. `sourceConfidence`
15. `sourceConfidenceReasons[]`
16. `isOfficialSource`
17. `notes`

这个层负责回答：**颜色到底从哪里来。**

### 7.2 `PaletteCluster`

表示一组相似颜色被聚类后的审核对象，核心字段建议包括：

1. `clusterId`
2. `batchId`
3. `occasionId`
4. `clusterName`
5. `seasonHint`
6. `palettePreview[]`
7. `styleTags[]`
8. `toneTags[]`
9. `evidenceIds[]`
10. `brandCoverageCount`
11. `officialEvidenceCount`
12. `duplicateRiskLevel`
13. `reviewStatus`
14. `reviewNotes`
15. `transferReadiness`

这个层负责回答：**这一组颜色值不值得保留。**

### 7.3 `ListingDraft`

表示准备转入正式 `palette` 的草稿，核心字段建议包括：

1. `draftId`
2. `clusterId`
3. `candidatePaletteName`
4. `occasionId`
5. `recommendedAudience`
6. `paletteColors[]`
7. `marketSignalSummary`
8. `referenceSources[]`
9. `editorialNotes`
10. `publishReadiness`

这个层负责回答：**是否可以进入正式 palette 上架链路。**

## 8. 采集策略升级建议

### 8.1 来源白名单分级

建议把来源从“能搜到就能用”改成分级策略：

#### A 级：可直接进入审核主链路

1. 品牌官网商品页
2. 天猫官方旗舰店商品页
3. 京东官方旗舰店商品页
4. 品牌官方电商页（公开可访问时）

#### B 级：可进入候选池，但需要人工确认

1. Net-a-Porter / Mytheresa / Farfetch 等可信多品牌平台商品页
2. 品牌官方 Lookbook / Campaign 页面

#### C 级：只能作为灵感参考，不直接进入上架链路

1. 搜索结果页
2. 社媒截图
3. 二次转载内容
4. 无法识别原始商品页的聚合内容

原则：

搜索结果页不能再作为正式候选来源本身，只能作为“发现入口”。

#### 访问边界

采样助手下一阶段不应假设“所有来源都可以自动访问”，而应明确三类访问边界：

1. **公开可访问网页**：不登录即可打开的官网商品页、官方旗舰店商品页、可信平台商品页，可以进入全自动采样链路。
2. **登录鉴权网页**：需要账号、会员态、地区态或其他授权才能访问的页面，不进入无人值守自动采样链路。系统不负责绕过鉴权，只接收人工协助后的证据。
3. **小程序页面**：不视为普通网页来源。只有在存在公开分享入口、对应 H5 页面或人工提交证据时，才允许进入采样链路。

这条边界很关键：

系统可以帮助整理证据、提取颜色、记录进度，但不应以“突破登录或权限限制”作为能力目标。

#### 采样模式

基于上面的访问边界，建议把来源采样模式拆成三种：

1. **全自动采样**：面向公开网页。系统自动发现链接、打开页面、抓取商品图、截图、提色并落库。
2. **半自动采样**：面向需要人工先提供入口的来源，例如小程序分享卡片、人工复制的真实商品页链接。系统接手后续证据整理、提色和聚类。
3. **人工证据采样**：面向只有截图或无法稳定访问的来源。运营上传截图、商品图、品牌、品类和观察时间，系统只负责证据归档、提色和后续审核，不把这类来源当作高可信自动来源。

#### 小程序处理原则

对于品牌官方小程序，建议按以下优先顺序处理：

1. 优先寻找对应官网商品页、天猫官方旗舰店页或京东官方旗舰店页，作为正式证据源。
2. 若小程序能稳定导出公开分享页或对应 H5 页面，可把该入口纳入半自动采样。
3. 若内容只存在于小程序内，且无法形成公开可访问链接，则走人工证据采样，并单独降低来源可信度等级。
4. 小程序证据可以作为灵感或补充，但不应默认成为正式 palette 上架的主证据源。

### 8.2 采集流程

建议把采集流程拆成 5 步：

1. 基于品牌白名单和场景矩阵发现候选链接
2. 落到真实商品页或真实展示页
3. 抓取页面标题、品牌、品类、商品图和截图
4. 对真实图像执行提色和去噪
5. 把结果落到 `SourceEvidence`

### 8.3 去重策略

至少要做三层去重：

1. URL 去重：同一个来源页只保留一条证据
2. 图像去重：相同或近似商品图判为同源
3. 色盘去重：综合色接近的结果进入同一个 `PaletteCluster`

只有去重之后，首页的色盘总览才会真正可用。

## 9. 审核到上架流程设计

建议把流程拆成以下六步：

### 9.1 第一步：来源证据入池

系统收集真实来源，输出 `SourceEvidence`，并打上可信度与来源类型标签。

### 9.2 第二步：自动提色与初步归类

系统从商品图或页面主视觉中提取主色、次色、点缀色，并按场景与色路初步分组。

### 9.3 第三步：自动聚类与重复检测

系统把相似颜色组合聚成 `PaletteCluster`，并标出：

1. 相似度高的重复组
2. 来源过少的弱样本
3. 来源不可信的待清理样本

### 9.4 第四步：人工审核聚类

运营主动作建议固定为：

1. `approve`：保留该色盘聚类
2. `merge`：并入相似聚类
3. `reject`：驳回该聚类
4. `hold`：暂缓观察

### 9.5 第五步：生成 `ListingDraft`

审核通过的聚类进入草稿层，自动带入：

1. 推荐名称
2. 代表色盘
3. 参考来源
4. 市场信号摘要
5. 场景与受众说明

### 9.6 第六步：正式 palette 上架审核

只有 `ListingDraft` 达到一定质量门槛，才进入现有 `palette` 正式链路。

建议门槛至少包括：

1. 至少 3 条有效来源证据
2. 至少覆盖 2 个品牌
3. 至少 1 条官方来源
4. 不与已发布 palette 高度重复
5. 有明确场景和目标用户解释

## 10. admin-web 侧的改版范围

### 10.1 页面层改造方向

`SamplingBatchesPage` 下一阶段不建议再以“批次列表 + 记录列表 + 右侧表单”为核心，而应改造成：

1. 批次切换栏
2. 顶部聚合指标栏
3. 色盘总览墙
4. 右侧证据抽屉
5. 次级人工修正面板

### 10.2 ViewModel 重心变化

当前 ViewModel 更偏向记录编辑；下一阶段应转向：

1. 批量筛选
2. 聚类选中态
3. 聚类审核动作
4. 证据抽屉状态
5. 合并 / 拆分操作
6. 转草稿动作

### 10.3 Transformer 重心变化

当前 transformer 更像把批次数据压成卡片摘要；下一阶段应增加：

1. 聚类展示模型
2. 证据展示模型
3. 指标统计模型
4. 重复风险和可信度标签转换

### 10.4 流式日志与任务透明度

如果采样助手保留“大模型分析 + 多步证据处理 + 自动聚类”的能力，那么它在产品形态上已经接近一个受控 Agent。对于这种模块，**流式日志不是附加装饰，而是任务透明度的基础设施**。

#### 可行性判断

流式日志是可行的，而且建议尽早纳入设计，原因如下：

1. 采样任务天然是多阶段过程，适合按步骤持续回传状态，而不是只在结束时给一个结果。
2. 运营非常关心“系统正在做什么、卡在哪、哪些来源被跳过、哪些需要人工接管”，这类信息不适合只藏在服务端控制台日志里。
3. 日志主要是服务端向前端的单向推送，技术上优先考虑 SSE（Server-Sent Events）；只有在未来需要双向控制时，再考虑 WebSocket。
4. 当前仓库里还没有现成的采样流式通道，说明这部分属于新增能力，但并不依赖复杂基础设施，作为 NestJS + React 的增量能力是可落地的。

#### 建议的任务模型

建议把每一次采样动作都显式建模成一个 `SamplingRun`，而不是让“生成候选”直接等同于一个黑盒接口调用。

`SamplingRun` 建议至少包含：

1. `runId`
2. `batchId`
3. `operationType`，例如 `discover-sources`、`collect-evidence`、`extract-colors`、`cluster-palettes`
4. `status`，例如 `queued`、`running`、`needsManualInput`、`succeeded`、`failed`、`cancelled`
5. `currentStage`
6. `progressPercent`
7. `startedAt`
8. `finishedAt`
9. `summary`
10. `warningCount`
11. `errorCount`

同时，每次运行应产生结构化的 `SamplingRunEvent` 序列，而不是只写纯文本字符串。

#### 建议的日志事件类型

建议把流式日志拆成结构化事件，至少覆盖：

1. `run-created`
2. `stage-started`
3. `candidate-link-found`
4. `page-opened`
5. `page-skipped`
6. `auth-required`
7. `manual-evidence-requested`
8. `screenshot-captured`
9. `image-selected`
10. `colors-extracted`
11. `cluster-created`
12. `cluster-merged`
13. `model-analysis-started`
14. `model-analysis-finished`
15. `warning`
16. `error`
17. `run-finished`

日志内容要做到“对运营有解释力”，而不是只有工程师才能看懂。例如：

1. 找到了哪些候选来源。
2. 哪些页面可以自动访问。
3. 哪些页面要求登录，已被降级为人工证据模式。
4. 系统最终选择哪张图作为提色主图。
5. 提出了哪些主色、次色、点缀色。
6. 哪些样本因重复被合并。
7. 模型分析是否启用、是否跳过、是否失败。

#### 传输方案建议

建议的优先顺序如下：

1. **首选 SSE**：适合服务端持续向前端单向推送日志和进度，前端实现成本低，浏览器原生支持 `EventSource`。
2. **补充查询接口**：保留 `GET /runs/:id` 和 `GET /runs/:id/events` 这类普通接口，用于页面刷新恢复、历史日志回放和非流式兜底。
3. **暂不优先 WebSocket**：只有在未来需要“前端主动暂停 / 恢复 / 注入人工输入 / 双向协作”时，再评估是否切到 WebSocket。

对当前 monorepo 而言，一个务实的首版形态是：

1. `POST /api/sampling-runs` 创建任务并返回 `runId`
2. `GET /api/sampling-runs/:id/stream` 以 `text/event-stream` 输出结构化日志
3. `GET /api/sampling-runs/:id` 返回当前摘要状态
4. `GET /api/sampling-runs/:id/events` 返回历史事件列表

#### admin-web 侧的日志呈现建议

前端建议把流式日志做成采样工作台里的一级能力，而不是开发期隐藏调试面板。页面上至少要有：

1. 当前阶段名称，例如“发现来源”“抓取商品图”“提取颜色”“聚类中”“模型分析中”
2. 总体进度条和阶段进度条
3. 时间线式日志列表
4. 警告和错误高亮
5. 需要人工介入时的显式提示
6. 运行完成后的摘要回顾

对于运营来说，最有价值的不是看到技术细节本身，而是看到“任务为什么停了”“哪一步失败了”“哪些来源被判定为低可信”“当前结果是否值得继续等”。

#### 与访问边界的联动

流式日志的另一个核心价值，是把“访问边界”透明化。系统不应默默跳过某些页面，而应明确记录：

1. 某个来源可公开访问，已自动采集。
2. 某个来源要求登录，已转人工证据模式。
3. 某个来源来自小程序，仅保留分享卡片或截图证据。
4. 某个来源因缺少商品图或截图质量过低，未进入提色阶段。

这样用户才能真正信任这套模块，而不是只看到一个最终色盘结果，却不知道它是怎么来的。

## 11. 分阶段实施建议

### Phase 1：重构数据合同与页面骨架

目标：先把页面工作对象从“记录”切到“聚类”。

交付：

1. 新的数据合同草案
2. `SamplingBatchesPage` 新信息架构骨架
3. 首页色盘总览墙占位
4. 证据抽屉占位
5. `SamplingRun` 基础结构与日志面板占位

### Phase 2：补真实来源证据链

目标：让候选不再只是模板结果，而是可追溯的真实来源。

交付：

1. `SourceEvidence` 数据落地
2. 真实商品页抓取结果接入
3. 页面截图 / 商品图 / 提色结果展示
4. 来源可信度标记
5. 基于 SSE 的运行中日志推送与任务摘要恢复

### Phase 3：补聚类审核与重复处理

目标：解决“各个色盘差不多”和“重复很多”的核心问题。

交付：

1. 自动聚类结果
2. 重复风险提示
3. 合并 / 驳回 / 暂缓操作
4. 批量审核能力

### Phase 4：接入 palette 草稿和上架闭环

目标：打通从采样助手到正式 palette 的链路。

交付：

1. `ListingDraft` 结构
2. 转写入口
3. 草稿审核页或草稿面板
4. 与正式 `palette.referenceSources[]` 的映射

## 12. 成功标准

改版后，采样助手应至少满足以下判断标准：

1. 运营进入页面后，3 分钟内能看清当前批次的主要色路分布。
2. 每个候选色盘都能追溯到真实来源页，而不是模糊搜索入口。
3. 同质化和重复样本显著下降，首页展示对象以聚类结果为主，不再是一长串碎片记录。
4. 运营的主动作从“补字段”转为“筛选、比较、审核、转草稿”。
5. 审核通过的色盘能顺畅转入正式 palette 草稿链路。
6. 运行中的采样任务可以实时看到当前阶段、进度、警告、错误和人工接管提示。

## 13. 当前建议的下一步

如果确认按这个方向推进，建议优先执行：

1. 先补一版新的采样数据合同草案，把 `SourceEvidence / PaletteCluster / ListingDraft` 结构明确下来。
2. 再重画 `SamplingBatchesPage` 的页面骨架，先把“全局总览 + 证据抽屉”立起来。
3. 最后替换当前模板式候选生成逻辑，逐步接入真实来源证据链。

在这三个步骤里，**第一优先级应是数据合同和页面主视角**。如果这两个不先改，后续继续往当前结构里堆自动生成能力，问题只会继续放大。
