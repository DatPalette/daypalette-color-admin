# DayPalette Color Admin 项目进度

> 记录日期：2026-05-07
> 记录目的：收口 `daypalette-color-admin` 当前可运行能力、已完成链路、现阶段缺口与下一步重点，便于后续继续开发或回看。

## 1. 当前状态一句话

`daypalette-color-admin` 当前已经从“基础资产管理后台”扩展到“真实资产管理 + 候选审阅台 + 自动候选生成”的阶段：

1. Base Colors / Dictionaries / Palettes / Collections 四类真实资源的本地读写链路已经跑通。
2. 管理台前端已经有独立的候选审阅台页面，可查看采样批次、逐条审阅候选并做少量修正。
3. `admin-api` 已提供采样批次文件读写、状态流转、自动候选生成与能力状态接口。
4. 自动候选生成已支持规则生成和可选模型深化两层模式。

## 2. 已完成内容

### 2.1 资产管理后台基础能力

已完成并保持可用：

1. `base-colors` 的读取、编辑、保存、新增、软删除、恢复与删除前引用检查。
2. `dictionaries` 的读取、保存、条目新增、软删除、恢复与引用检查。
3. `palettes` 的读取、编辑、保存、新增、软删除、恢复与引用检查。
4. `collections` 的读取、编辑、成员排序、封面调整、保存、软删除、恢复与引用检查。
5. 管理台外壳、路由、共享展示组件与格式化 helper 的基础收口。

### 2.2 采样批次与候选审阅链路

已完成：

1. 在共享 contracts 中补齐 `SamplingBatchDocument`、`SamplingRecord`、批次状态、记录状态和能力状态类型。
2. 在 `packages/file-store` 中补齐采样批次文件路径解析。
3. 在 `admin-api` 中新增采样批次文件读写层：`data/palette-sampling/<batchId>.v1.json`。
4. 在 `admin-api` 中新增采样批次模块，提供以下接口：
   - `GET /api/sampling-batches`
   - `GET /api/sampling-batches/:id`
   - `GET /api/sampling-batches/capabilities`
   - `POST /api/sampling-batches/:id/generate-candidates`
   - `PUT /api/sampling-batches/:id`
   - `PATCH /api/sampling-batches/:id/status`
   - `DELETE /api/sampling-batches/:id`
5. 已把采样助手接入前端工作台路由：`/sampling-batches`。
6. 已完成候选审阅台主页面、批次列表、记录导航、逐条审阅、保存与删除动作。
7. 已把页面主流程收敛成“生成候选 -> 逐条审阅 -> 必要时少量修正”。
8. 已增加“近似色盘预览”，可用主色/次色/点缀色摘要做语义色块预览。

### 2.3 自动候选生成能力

已完成：

1. 后端规则生成支持按 `themeKey`、品牌池、品类与渠道类型输出候选品牌、平台、来源链接、观察日期、颜色摘要、风格信号与市场信号。
2. 候选生成能力状态已可被前端读取，并在页面上显示“规则生成模式”或“AI 深化已启用”。
3. 前端按钮已经从单纯“生成”改到“刷新当前批次候选”的审阅语义。
4. 规则生成现在不再为同一主题只套一套固定色路模板，而是按记录稳定分配不同色路变体，减少 24 条候选中整组重复的问题。
5. 对于仍处于 `sampled` 且仍属于系统候选的记录，重新生成时会刷新自动生成字段，而不是因为已有值就保持不变。

### 2.4 模型配置支持

已完成：

1. `admin-api` 已支持通过 `apps/admin-api/.env` 读取模型配置。
2. 已引入 `dotenv` 并在启动入口自动加载环境变量。
3. 已提供 `apps/admin-api/.env.example` 示例文件。
4. 当前支持以下环境变量：
   - `DAYPALETTE_LLM_API_KEY`
   - `DAYPALETTE_LLM_MODEL`
   - `DAYPALETTE_LLM_BASE_URL`
5. 已补充 Xiaomi Mimo 与千问（DashScope OpenAI 兼容接口）的示例配置说明。

## 3. 当前已知边界

当前仍然没有完成的能力：

1. 还没有真实网页抓取与页面解析，当前候选仍以规则生成和模型摘要为主。
2. 还没有把候选自动转写成正式 `palette.referenceSources[]` 或正式 palette 草稿。
3. 还没有批量审阅、批量通过/驳回与批量重生成。
4. 还没有更严格的“24 条候选内颜色组合完全去重”约束，目前只是显著降低重复，不是数学意义上的全量唯一。
5. 还没有真正回溯到需要登录的平台详情页做稳定提取，因此候选来源链接目前更偏“公开搜索入口 / 品牌官网入口”，而不是承诺已经提取到了平台登录后内容。

## 4. 这次迭代解决的关键问题

这次迭代重点解决了以下问题：

1. 页面过于像“说明页”而不是“审阅台”，现在主界面已经明显压缩说明区，审阅内容占主场。
2. 候选色路按主题整组重复，导致“审阅没有意义”，现在规则生成已经改为同主题多变体分配。
3. 页面之前没有任何色盘感知，现在已提供语义近似色盘预览。
4. 之前的天猫链接会给人“系统抓到了真实颜色”的错觉，但实际并没有；当前已经改成更诚实的公开检索/官网入口表达。
5. 模型接入之前只能靠终端环境变量，现在已经有稳定的 `.env` 配置方式。

## 5. 当前验证状态

本轮已验证：

1. `pnpm --dir apps/admin-api build` 通过。
2. `pnpm --dir apps/admin-web build` 通过。

说明：当前验证覆盖了共享包构建、后端构建和前端构建；未包含真实外部来源抓取验证，因为该能力尚未接入。

## 6. 下一步建议

优先顺序建议如下：

1. 为候选生成补“同批次全量去重约束”，确保 24 条候选不再出现肉眼可见的重复色路。
2. 将公开搜索入口进一步提升到更可靠的品牌详情页 / 商品页候选，而不是停留在搜索页。
3. 增加“候选转正式引用源”动作，把审阅通过记录映射到 `palette.referenceSources[]` 草稿。
4. 增加批量重生成、批量通过和批量驳回，减少人工点选成本。
5. 若要进一步降低人工成本，再评估接入真实网页抓取、截图理解或更强的来源解析链路。
